## What went well

- Verified the nix wiring with a real `nix build` of the activation package
  and inspected the built `home-manager-files` output for the actual symlink,
  instead of assuming `recursive = true` picks up a new sibling dir. This is
  the standing "ground truth beats reasoning" lesson, and it also settled a
  dead end: per-file `nix eval` of `home.file.".claude/skills/<x>"` fails
  because the recursive source registers one dir-level entry, so the build was
  the right probe, not the eval.
- Used an independent subagent for the review rather than self-reviewing. It
  caught the real MAJOR (the spike/plan boundary was self-contradictory) that
  a self-review would have been tempted to rationalize away. The review skill's
  "fresh, skeptical eyes" is worth taking literally even for a docs change.
- Grounded the fix in an existing convention (tatr's rule that a stepless
  ad-hoc task is planned first with `/plan`, tatr/SKILL.md:70-71) instead of
  inventing a new one, so spike now composes with plan cleanly.

## What went wrong

- Two `tatr new` calls in the same shell line got the same second-resolution
  ID and the second silently overwrote the first. Root cause: tatr IDs are
  `YYYYMMDD-HHMMSS`, one-second granular, and I fired both creations in one
  fast command. Recovered by hand-creating the first task dir with a distinct
  timestamp.
- The spike/plan contradiction shipped into review round 1. Root cause: the
  Relationship section was written aspirationally ("planned and built") without
  reconciling it against Step 5, which had spike create the tasks itself. The
  two sections were authored in different passes and never cross-checked.

## What to improve next time

- When creating several tatr tasks at once, hand-assign distinct IDs (or space
  the calls) - do not rely on `tatr new` in a tight loop, its IDs collide
  within a second.
- For a self-referential workflow skill, dry-run the lifecycle claims against
  each other before handing to review: read the Relationship section and every
  Step as if executing them in sequence, and check they describe one model.
  This is the same "mental dry-run of the new instructions" lesson from the
  squash-merge retro, applied to internal consistency rather than git
  mechanics.
