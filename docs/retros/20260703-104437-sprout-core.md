## What went well

- Verified the Nix indented-string dollar escaping empirically (`nix eval` +
  `od -c`) instead of guessing, which confirmed the brace-free `$VAR` style
  from `tmux-sessionizer.nix` needs almost no escaping. Cheap experiment,
  removed a whole class of build-time guesswork.
- Reviewing with adversarial inputs (`../escape`, a slash-named feature,
  outside-a-repo) rather than just the happy path is what surfaced the MAJOR
  bug. The happy-path manual test during `/work` had passed clean.
- Building just the package via the flake's nixpkgs
  (`nix build --impure --expr 'head (import ./sprout.nix ...).home.packages'`)
  gave a fast shellcheck + smoke loop without a full home-manager rebuild.

## What went wrong

- R1.1 (MAJOR): `cmd_new` printed the path and exited 0 even when
  `git worktree add` failed. Root cause: the script runs under `set +o
  errexit` (repo house style), so a failing command does **not** abort - but I
  wrote `new` as if it would, never checking the add's result. The stdout
  contract ("print the path so callers can `cd`") was also decided after the
  fact rather than up front, so nothing forced me to gate the `echo` on
  success.
- R1.2/R1.3 were latent because the feature name flowed straight into a path
  with no validation and slash-name cleanup was never considered - both are
  "I only tested `feat-a`" gaps, i.e. too-narrow test inputs during `/work`.
- Process friction: two `tatr new` calls in the same second produced the same
  timestamp ID and the second clobbered the first. Had to split the dirs by
  hand. Not a code bug but it cost a detour.

## What to improve next time

- In any shell tool with `set +o errexit`, treat every fallible external
  command as needing an explicit success check (`if ! cmd; then ...`), and
  decide each command's stdout/exit contract before writing it, not after.
- During `/work`, test adversarial inputs (traversal, slashes, empty, flags)
  and the actual downstream use (`$(...)` capture), not just one happy value -
  that is exactly what the reviewer will do.
- When creating several tatr tasks at once, space the `tatr new` calls or
  assign distinct timestamp dirs to avoid same-second ID collisions.
