## What went well

- One review round to APPROVE. The core/tmux split from planning paid off:
  because task 1 kept worktree logic free of any tmux coupling, this task was
  purely additive (new helpers + flag parsing), nothing in the core had to be
  unpicked. That is the plan boundary doing its job.
- Carried the previous retro's lesson forward on purpose: the one fallible new
  command (`tmux new-session`) is guarded and exits non-zero on failure, while
  has-session/kill-session errors are deliberately tolerated. The lesson
  compounded instead of being re-learned via a review finding.
- Isolated tmux tests with `TMUX_TMPDIR` so the sandbox never touched a real
  tmux server - the right instinct given the user lives in tmux.

## What went wrong

- No code finding against the diff, but the review nearly misreported a fact:
  a test measured `$?` after `"$SPROUT" ... | sed ...`, so it captured sed's
  exit (0), not sprout's, and briefly looked like `-i new` on an existing
  worktree returned 0. Caught on inspection. Root cause: piping a command into
  a formatter before checking its exit code.
- The review surfaced a genuine pre-existing wart (project derived from the
  current worktree, not the main one). Not introduced here, but it means the
  tool misbehaves from inside a worktree - exactly where a parallel agent
  runs. Correctly routed to a follow-up task rather than scope-creeping this
  branch.

## What to improve next time

- Never read `$?` through a pipe. Capture output to a variable (or use
  `PIPESTATUS`) and check the real command's exit code.
- When a review finding's root cause lives in an earlier task, file it as its
  own tatr task immediately (done) and keep the current branch focused.
